Service Communication

FromToProtocolPurpose
FrontendBackendREST/HTTPUser actions (auth, booking, queries)
BackendRabbitMQAMQPPublish appointment events
RabbitMQPython ServiceAMQPDeliver events for processing
Python ServiceBackendREST/HTTPUpdate appointment status (callback)

Design Principles

  • Event-Driven Architecture — Loose coupling between services via message broker
  • Stateless Authentication — JWT tokens, no server-side sessions
  • Layered Architecture — Controller → Service → Repository pattern
  • Single Responsibility — Each service has one clear purpose
  • Fail-Safe Messaging — Durable queues, manual ACK, retry logic

6. Authentication & Security

JWT Authentication Flow

1. Client sends POST /api/auth/login with {email, password} 2. Server validates credentials via AuthenticationManager + BCrypt 3. Server generates JWT token (HS384, 24h expiry) 4. Client stores token in localStorage 5. Client sends token in every request: Authorization: Bearer <token> 6. JwtAuthFilter intercepts request, validates token, sets SecurityContext 7. Controller method accesses user via Authentication parameter

Security Configuration

FeatureImplementation
Password HashingBCrypt (Spring Security default)
Token AlgorithmHMAC-SHA384 via jjwt library
Token Expiry24 hours (86400000 ms)
Session ManagementStateless (no server-side sessions)
CORSAllowed origins: localhost:3000, localhost:5173
CSRFDisabled (stateless API with JWT)
Service-to-ServiceShared API key via X-Internal-Key header

Public vs Protected Endpoints

Endpoint PatternAccess
/api/auth/**🟢 Public
/api/internal/**🟢 Public (but API-key protected)
/swagger-ui/**🟢 Public
/h2-console/**🟢 Public
/actuator/**🟢 Public
All other endpoints🔴 JWT Required

Security Measures

  • SQL Injection Prevention — JPA parameterized queries, no raw SQL
  • Input Validation — Bean validation annotations on all DTOs (@NotBlank, @Email, @Size)
  • Authorization — Users can only access/cancel their own appointments
  • Exception Handling — GlobalExceptionHandler prevents stack trace leakage
  • Password Security — Never returned in API responses

RabbitMQ Configuration

ComponentNameType
Exchangeappointment.eventsTopic Exchange (durable)

Routing Keyappointment.createdBinds to created queue

Event Payload

{
  "eventType": "APPOINTMENT_CREATED",       // or "APPOINTMENT_CANCELLED"
  "appointmentId": 1,
  "patientName": "John Doe",
  "patientEmail": "john@example.com",
  "doctorName": "Dr. Sarah Johnson",
  "date": "2026-05-23",
  "startTime": "09:00",
  "endTime": "09:30",
  "timestamp": "2026-05-23T10:15:30"
}

Message Flow

Spring Boot RabbitMQ Python Service │ │ │ │ 1. Publish event │ │ │─────────────────────────────▶│ │ │ (JSON via Jackson2Json) │ │ │ │ 2. Deliver to consumer │ │ │─────────────────────────────▶│ │ │ (manual ACK, prefetch=1) │ │ │ │ │ │ │ 3. Process event │ │ │ (simulate email) │ │ │ │ 4. Status callback │ │ │◀─────────────────────────────│──────────────────────────────│ │ PUT /api/internal/... │ │ │ (X-Internal-Key auth) │ │ │ │ │ │ │ 5. ACK message │ │ │◀─────────────────────────────│ │ │ │

Reliability Features

  • Durable Queues — Messages survive RabbitMQ restart
  • Manual ACK — Messages only removed after successful processing
  • NACK + Requeue — Failed messages are requeued for retry
  • Prefetch Count = 1 — Process one message at a time
  • Connection Retry — Python service retries connection up to 10 times
  • JSON Serialization — Jackson2JsonMessageConverter for type-safe messaging

8. Python Notification Service

Architecture

FilePurpose
main.pyEntry point, RabbitMQ connection + consumer loop
notification_handler.pyEvent processing, email simulation
callback_client.pyREST client to update appointment status

Processing Pipeline

  1. Receive message from RabbitMQ queue
  2. Parse JSON payload
  3. Call Spring Boot API: status → NOTIFICATION_SENDING
  4. Process event:
    • APPOINTMENT_CREATED → Log formatted booking confirmation email
    • APPOINTMENT_CANCELLED → Log formatted cancellation notice
  5. Simulate email sending delay (2 seconds)
  6. Call Spring Boot API: status → NOTIFICATION_SENT
  7. ACK the RabbitMQ message

Simulated Email Output

SENDING EMAIL NOTIFICATION - Appointment Confirmation
============================================================

  ---
  Dear John Doe,

  Your appointment has been confirmed!

    Appointment ID: #1

  Please arrive 10 minutes early.

  Best regards,
  Healthcare Platform Team
============================================================

Error Handling

  • Messages that fail processing are NACKed and requeued
  • Connection failures trigger automatic retry (10 attempts, 5s delay)
  • HTTP callback failures are logged but don't block message processing
  • Graceful shutdown on KeyboardInterrupt (SIGINT)

12. Concurrency & Duplicate Prevention

Multi-Layer Protection

LayerMechanismProtection
Applicationslot.isAvailable checkFirst check — prevents most duplicates
ApplicationexistsBySlotIdAndStatusNot(CANCELLED)Secondary check in repository
DatabaseUNIQUE constraint on appointments.slot_idFinal guarantee — DB-enforced uniqueness
Application@Version on Appointment entityOptimistic locking for concurrent updates
Transaction@Transactional on service methodsAtomic read-modify-write operations

Check slot.isAvailable │ │ │ │ 2. Check existsBySlotId │ │ │ │ 3. Set slot.available=false │ │ │ │ 4. INSERT appointment │ │ │ │ │ │ │ │ Winner: First to INSERT wins │ │ │ │ Loser: UNIQUE violation │ │ │ │ → 409 Conflict │ │ │ └───────────────┬───────────────┘ │ │ │ │ │ 200 OK: Booked! │ 409: Slot already booked │ │◀─────────────────────────────│─────────────────────────────▶│

14. Testing Guide

Manual Test Scenarios

Test 1: User Registration & Login

  1. Open http://localhost:3000
  2. Click "Register" → fill in name, email, password → Submit
  3. Verify: Redirected to Dashboard
  4. Logout → Login with same credentials
  5. Verify: Dashboard shows "Hello, [name]"

Test 2: Appointment Booking (Happy Path)

  1. Click "Book Appointment"
  2. Select a doctor → Select today's date → Pick an available slot
  3. Click "Confirm Booking"
  4. Verify visual flow: "Booking appointment..." → "Appointment booked successfully!" → "Processing notification event..." → "All done!"
  5. Click "View My Appointments" → Verify appointment appears with status progressing from CONFIRMED → NOTIFICATION_SENT
  6. Check Python service logs for simulated email output

Test 3: Duplicate Booking Prevention

  1. Note the slot that was just booked
  2. Try to book the same slot again (via another browser/API call)
  3. Verify: Error "Slot is no longer available" (409 Conflict)

Test 4: Appointment Cancellation

  1. Go to "My Appointments"
  2. Click "Cancel" on an active appointment → Confirm
  3. Verify: Status changes to "Cancelled"
  4. Verify: The slot becomes available again for new bookings
  5. Check Python service logs for cancellation email

Test 5: Authorization

  1. Register a second user in a different browser/incognito
  2. Try to cancel the first user's appointment via API
  3. Verify: 403 "You can only cancel your own appointments"

Test 6: API via Swagger

  1. Open http://localhost:8080/swagger-ui.html
  2. Try register/login → Copy the JWT token
  3. Click "Authorize" → Paste token
  4. Test all endpoints directly

Test 7: RabbitMQ Monitoring

  1. Open http://localhost:15672 (guest/guest)
  2. Navigate to Queues tab
  3. Verify: appointment.created.queue and appointment.cancelled.queue exist
  4. Book an appointment → Watch message appear and get consumed

cURL Test Commands

# Register
curl -X POST http://localhost:8080/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"fullName":"Test User","email":"test@example.com","password":"password123"}'

# Login
curl -X POST http://localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"password123"}'

# Get Doctors (use token from login response)
curl http://localhost:8080/api/slots/doctors \
  -H "Authorization: Bearer YOUR_JWT_TOKEN"

# Get Available Slots
curl "http://localhost:8080/api/slots?doctorId=1&date=2026-05-23" \
  -H "Authorization: Bearer YOUR_JWT_TOKEN"

# Book Appointment
curl -X POST http://localhost:8080/api/appointments \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_JWT_TOKEN" \
  -d '{"slotId": 1}'

# Get My Appointments
curl http://localhost:8080/api/appointments \
  -H "Authorization: Bearer YOUR_JWT_TOKEN"

# Cancel Appointment
curl -X DELETE http://localhost:8080/api/appointments/1 \
  -H "Authorization: Bearer YOUR_JWT_TOKEN"
